# **常見問題：代理式設計模式**

## 1. 什麼是「代理式設計模式」？

代理式設計模式是一種可重複使用的高層次解法，用來處理建構智慧自主系統（代理（agent））時常見的問題。這些模式提供一套結構化框架，用來設計代理行為，就像軟體設計模式之於傳統程式設計一樣。它們能協助開發者建立更穩健、可預測且有效的 AI 代理。

## 2. 本指南的主要目標是什麼？

本指南旨在提供一份實務且動手操作導向的入門內容，協助讀者設計與建構代理式系統。它不只停留在理論討論，而是提供具體的架構藍圖，讓開發者能以可靠的方式建立具備複雜、目標導向行為的代理。

## 3. 本指南的目標讀者是誰？

本指南是為正在使用大型語言模型（large language model, LLM）與其他 AI 元件建構應用程式的 AI 開發者、軟體工程師與系統架構師而寫。它適合希望從簡單的提示與回應互動，進一步建立精密自主代理的人。

## 4. 本指南討論了哪些關鍵代理式模式？

根據目錄，本指南涵蓋數個關鍵模式，包括：

- **反思（Reflection）：** 代理批判自身行動與輸出的能力，用以改善效能。  
- **規劃（Planning）：** 將複雜目標分解成較小且可管理的步驟或任務的過程。  
- **工具使用（Tool Use）：** 代理使用外部工具（例如程式碼直譯器、搜尋引擎或其他 API）來取得資訊，或執行它無法獨力完成的動作。  
- **多代理協作：** 讓多個專門代理共同解決問題的架構，通常會涉及一個「leader」或「orchestrator」代理。  
- **人類介入（Human in the Loop）：** 整合人類監督與介入，讓人類能對代理的行動提供回饋、修正與核准。

## 5. 為什麼「規劃」是重要模式

規劃之所以重要，是因為它能讓代理處理無法透過單一動作解決的複雜多步驟任務。透過建立計畫，代理可以維持一致的策略、追蹤進度，並以結構化方式處理錯誤或意外障礙。這能避免代理「卡住」或偏離使用者的最終目標。

## 6. 對代理而言，「工具」和「技能」有什麼差異

雖然這兩個詞經常交替使用，但「工具」通常指代理可以呼叫的外部資源（例如天氣 API、計算機）。「技能」則是代理已學會、更加整合的能力，通常會結合工具使用與內部推理來執行特定功能（例如「預訂航班」這項技能，可能會涉及使用行事曆與航空公司 API）。

## 7. 「反思」模式如何改善代理效能

反思是一種自我修正形式。代理在產生回應或完成任務後，可以被提示去檢視自己的工作、檢查錯誤、根據特定標準評估品質，或思考替代方法。這種反覆精煉過程能協助代理產生更準確、更相關且品質更高的結果。

## 8. 反思模式的核心概念是什麼

反思模式讓代理具備退一步批判自己工作的能力。代理不是一次就產出最終結果，而是先產生草稿，接著對草稿進行「反思」，找出缺陷、缺漏資訊或可改進之處。這種自我修正過程，是提升回應品質與準確性的關鍵。

## 9. 為什麼簡單的「提示鏈（Prompt Chaining）」不足以產生高品質輸出？** 

簡單提示鏈，也就是將一個提示的輸出作為下一個提示的輸入，通常過於基礎。模型可能只是改寫先前的輸出，而沒有真正改善內容。真正的反思模式需要更結構化的批判，提示代理根據特定標準分析自己的工作、檢查邏輯錯誤，或驗證事實。

## 10. 本章提到的兩種主要反思類型是什麼？** 

本章討論兩種主要的反思形式：

- **「檢查你的工作」反思：** 這是一種基本形式，只是要求代理檢視並修正先前的輸出。它是捕捉簡單錯誤的良好起點。  
- **「內部批判者」反思：** 這是較進階的形式，會使用另一個「critic」代理（或專用提示）來評估「worker」代理的輸出。這個批判者可以被賦予特定檢查標準，進而帶來更嚴謹且更有針對性的改進。

## 11. 反思如何協助減少「幻覺」？

透過提示代理檢視自己的工作，特別是將其陳述與已知來源比較，或檢查自身推理步驟，反思模式可以顯著降低幻覺（編造事實）的可能性。代理會被迫更扎根於提供的上下文，也較不容易產生缺乏支撐的資訊。

## 12. 反思模式可以套用不只一次嗎？

可以，反思可以是反覆進行的過程。代理可以多次反思自己的工作，每一輪都進一步精煉輸出。這對複雜任務特別有用，因為第一次或第二次嘗試可能仍包含細微錯誤，或還有大幅改善的空間。

## 13. 在 AI 代理的脈絡中，規劃模式是什麼？

規劃模式涉及讓代理能將複雜、高層次目標分解為一連串較小且可執行的步驟。代理不是試圖一次解決大問題，而是先建立「計畫」，再逐一執行計畫中的每個步驟；這是一種可靠得多的方法。

## 14. 為什麼複雜任務需要規劃？

LLM 在需要多個步驟或相依性的任務上可能會遇到困難。若沒有計畫，代理可能會忘記整體目標、漏掉關鍵步驟，或無法將某一步的輸出作為下一步的輸入。計畫提供清楚的路線圖，確保原始請求的所有需求都能依照邏輯順序被滿足。

## 15. 實作規劃模式的常見方式是什麼？

常見的實作方式，是讓代理先以結構化格式（例如 JSON array 或編號清單）產生步驟清單。系統接著可以逐一迭代這份清單，依序執行每個步驟，並將結果回饋給代理，以告知下一個動作。

## 16. 代理在執行期間如何處理錯誤或變更？

穩健的規劃模式允許動態調整。如果某個步驟失敗或情況改變，代理可以被提示從目前狀態重新規劃。它可以分析錯誤、修改剩餘步驟，甚至新增步驟來克服障礙。

## 17. 使用者會看到計畫嗎？

這是一項設計選擇。在許多情況下，先向使用者顯示計畫並取得核准，是很好的做法。這符合人類介入模式，能在代理執行建議動作之前，讓使用者擁有透明度與控制權。

## 18. 「工具使用」模式包含什麼？

工具使用模式允許代理透過與外部軟體或 API 互動來擴展能力。由於 LLM 的知識是靜態的，也無法自行執行真實世界的動作，因此工具能讓它存取即時資訊（例如 Google Search）、專有資料（例如公司的資料庫），或具備執行動作的能力（例如寄送電子郵件、預訂會議）。

## 19. 代理如何決定要使用哪個工具？

代理通常會取得一份可用工具清單，並附上每個工具的用途描述與所需參數。當面對無法用內部知識處理的請求時，代理的推理能力會讓它從清單中選出最合適的工具來完成任務。

## 20. 此脈絡中提到的「ReAct」（Reason and Act）框架是什麼？

ReAct 是一個常見框架，整合了推理與行動。代理會遵循 **Thought**（推理自己需要做什麼）、**Action**（決定要使用哪個工具以及使用哪些輸入）與 **Observation**（查看工具回傳結果）的迴圈。這個迴圈會持續進行，直到代理收集到足夠資訊來滿足使用者請求。

## 21. 實作工具使用有哪些挑戰？

關鍵挑戰包括：

- **錯誤處理：** 工具可能會失敗、回傳非預期資料，或發生逾時。代理需要能辨識這些錯誤，並決定要重試、改用其他工具，或向使用者求助。  
- **安全性：** 讓代理存取工具，特別是能執行動作的工具，會帶來安全性影響。對敏感操作設置防護措施、權限控管，以及通常需要人類核准，都非常重要。  
- **提示：** 必須有效提示代理，使其產生格式正確的工具呼叫（例如正確的函式名稱與參數）。

## 22. 什麼是人類介入（HITL）模式？

人類介入是一種將人類監督與互動整合到代理工作流程中的模式。代理不會完全自主運作，而是在關鍵節點暫停，要求人類提供回饋、核准、釐清或指示。

## 23. 為什麼人類介入對代理式系統很重要？

它之所以重要，有幾個原因：

- **安全與控制：** 對於高風險任務（例如金融交易、寄送正式通訊），人類介入能確保代理提出的動作在執行前先由人類驗證。  
- **改善品質：** 人類可以提供修正或細緻回饋，讓代理用來改善效能，尤其是在主觀或模糊任務中。  
- **建立信任：** 使用者更可能信任並採用他們可以引導與監督的 AI 系統。

## 24. 在工作流程中的哪些時間點應納入人類？

常見的人類介入時間點包括：

- **計畫核准：** 在執行多步驟計畫之前。  
- **工具使用確認：** 在使用會產生真實世界後果或花費金錢的工具之前。  
- **模糊性解決：** 當代理不確定如何繼續，或需要使用者提供更多資訊時。  
- **最終輸出審查：** 在將最終結果交付給終端使用者或系統之前。

## 25. 持續的人類介入不會很沒效率嗎？

可能會，因此關鍵在於找到適當平衡。人類介入應該實作在關鍵檢查點，而不是每個單一動作。目標是在人類與代理之間建立協作夥伴關係，由代理處理大部分工作，人類則提供策略性指引。

## 26. 什麼是多代理協作模式？

這個模式涉及建立一個由多個專門代理組成的系統，讓它們共同達成同一目標。與其讓一個「generalist」代理試圖完成所有事情，不如建立一支「specialist」代理團隊，每個代理都有特定角色或專業能力。

## 27. 多代理系統有哪些好處？

- **模組化與專業化：** 每個代理都可以針對其特定任務進行微調與提示（例如「researcher」代理、「writer」代理、「code」代理），進而產生更高品質的結果。  
- **降低複雜度：** 將複雜工作流程拆解成專門角色，能讓整體系統更容易設計、偵錯與維護。  
- **模擬腦力激盪：** 不同代理可以針對同一問題提供不同觀點，帶來更具創意且更穩健的解法，類似人類團隊的運作方式。

## 28. 多代理系統的常見架構是什麼？

常見架構會包含一個 **Orchestrator Agent**（有時稱為「manager」或「conductor」）。協調代理會理解整體目標、將其分解，並把子任務委派給合適的專門代理。接著，它會收集專家的結果，並將其綜合成最終輸出。

## 代理之間如何溝通？

溝通常由協調代理管理。例如，協調代理可能會將「researcher」代理的輸出作為上下文傳給「writer」代理。另一種常見溝通方式，是使用共用的「scratchpad」或 message bus，讓代理可以張貼自己的發現。

## 為什麼評估代理比評估傳統軟體程式更困難？

傳統軟體具有決定性輸出（相同輸入永遠產生相同輸出）。代理，特別是使用 LLM 的代理，則具有非決定性，而且其效能可能帶有主觀性。評估代理需要評估其輸出的 *品質* 與 *相關性*，而不只是判斷它在技術上是否「正確」。

## 評估代理效能有哪些常見方法？

本指南建議幾種方法：

- **以結果為基礎的評估：** 代理是否成功達成最終目標？例如，如果任務是「預訂航班」，航班是否真的被正確預訂？這是最重要的衡量方式。  
- **以流程為基礎的評估：** 代理的 *流程* 是否有效率且合乎邏輯？它是否使用了正確工具？是否遵循合理計畫？這有助於偵錯代理失敗的原因。  
- **人類評估：** 讓人類根據有用性、準確性與一致性等標準，以量表（例如 1-5 分）評估代理效能。這對面向使用者的應用程式至關重要。

## 什麼是「代理軌跡」？

代理軌跡是代理執行任務時所有步驟的完整紀錄。它包含所有思考、動作（工具呼叫）與觀察。分析這些軌跡，是偵錯與理解代理行為的關鍵部分。

## 如何為非決定性系統建立可靠測試？

雖然你無法保證代理輸出的確切措辭，但可以建立檢查關鍵元素的測試。例如，你可以撰寫測試來驗證代理的最終回應是否 *包含* 特定資訊，或它是否使用正確參數成功呼叫某個工具。這通常會在專用測試環境中使用 mock tools 完成。

## 提示代理和簡單的 ChatGPT 提示有何不同？

提示代理涉及建立詳細的「system prompt」或 constitution，作為其操作指令。這不只是單一使用者查詢；它會定義代理的角色、可用工具、應遵循的模式（例如 ReAct 或規劃）、限制，以及個性。

## 良好代理 system prompt 的關鍵組成是什麼？

強大的 system prompt 通常包含：

- **角色與目標：** 清楚定義代理是誰，以及其主要目的。  
- **工具定義：** 可用工具清單、其描述，以及如何使用它們（例如以特定 function-calling 格式）。  
- **限制與規則：** 明確指示代理 *不應該* 做什麼（例如「未經核准不得使用工具」、「不得提供財務建議」）。  
- **流程指示：** 關於應使用哪些模式的指引。例如：「先建立計畫。接著逐步執行計畫。」  
- **範例軌跡：** 提供幾個成功的「thought-action-observation」迴圈範例，可以顯著提升代理的可靠性。

## 什麼是「提示洩漏」？

提示洩漏是指 system prompt 的某些部分（例如工具定義或內部指令）被不慎揭露在代理給使用者的最終回應中。這可能讓使用者困惑，也會暴露底層實作細節。使用不同提示分別處理推理與產生最終答案等技巧，有助於避免這種情況。

## 代理式系統有哪些未來趨勢？

本指南指出未來將包含：

- **更自主的代理：** 需要較少人類介入，並且能自行學習與適應的代理。  
- **高度專門化的代理：** 可針對特定任務被雇用或訂閱的代理生態系（例如旅行代理、研究代理）。  
- **更好的工具與平台：** 發展更精密的框架與平台，使建構、測試與部署穩健多代理系統變得更容易。
