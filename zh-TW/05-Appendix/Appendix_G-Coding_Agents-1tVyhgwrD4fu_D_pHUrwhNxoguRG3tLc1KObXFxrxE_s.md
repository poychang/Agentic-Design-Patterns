# 附錄 G - 程式碼代理

## Vibe Coding：起點

「Vibe coding」已成為快速創新與創意探索的強大技術。這種做法使用 LLM 產生初始草稿、勾勒複雜邏輯，或建立快速 prototype，大幅降低初期阻力。它對克服「空白頁」問題非常有價值，讓開發者能快速從模糊概念前進到具體且可執行的程式碼。Vibe coding 在探索不熟悉的 API 或測試新的架構模式時特別有效，因為它繞過了一開始就需要完美實作的要求。產生的程式碼通常會成為創意催化劑，提供一個基礎，讓開發者能批判、重構並擴充。它的主要優勢在於能加速軟體生命週期早期的探索與構思階段。然而，雖然 vibe coding 擅長腦力激盪，開發穩健、可擴充且可維護的軟體仍需要更有結構的方法，也就是從純粹產生轉向與專門程式碼代理的協作夥伴關係。

## 代理作為團隊成員

最初的浪潮聚焦於原始程式碼產生——適合構思的「vibe code」——但業界現在正轉向更整合、更強大的生產工作典範。最有效的開發團隊不只是把任務委派給代理（agent）；他們正在用一組精密的程式碼代理強化自身能力。這些代理就像不知疲倦的專門團隊成員，放大人類創造力，並大幅提升團隊的可擴充性與速度。

這項演進也反映在業界領導者的說法中。2025 年初，Alphabet CEO Sundar Pichai 指出，在 Google，**「***over 30% of new code is now assisted or generated by our Gemini models, fundamentally changing our development velocity.*  Microsoft 也提出了類似說法。這項全產業轉變顯示，真正的前沿不是取代開發者，而是賦能開發者。目標是一種增強式關係：人類負責引導架構願景與創意問題解決，而代理處理測試、文件與 review 等專門且可擴充的任務。

本章提出一個用來組織人類－代理團隊的框架，其核心理念是人類開發者擔任創意負責人與架構師，而 AI 代理作為力量倍增器。此框架建立在三項基礎原則之上：

1. **人類主導的協調：** 開發者是團隊負責人與專案架構師。他們始終在人類介入中，協調工作流程、設定高階目標，並做出最終決策。代理很強大，但它們是支援性的協作者。開發者決定要啟用哪個代理、提供必要上下文，最重要的是對任何代理產生的輸出行使最終判斷，確保它符合專案品質標準與長期願景。  
2. **上下文優先：** 代理的表現完全取決於上下文的品質與完整性。強大的 LLM 若缺乏良好上下文，就毫無用處。因此，我們的框架優先採取由人類主導、細緻的上下文策展方法。應避免自動化、黑箱式的上下文擷取。開發者負責為其代理團隊成員組裝完美的「briefing」。這包括：  
   * **完整程式碼庫：** 提供所有相關 source code，讓代理理解既有模式與邏輯。  
   * **外部知識：** 提供特定文件、API 定義或設計文件。  
   * **人類簡報：** 清楚表達目標、需求、pull request descriptions 與風格指南。  
3. **直接模型存取：** 為了達到最先進成果（state-of-the-art results），代理必須由可直接存取前沿模型（frontier models，例如 Gemini 2.5 PRO、Claude Opus 4、OpenAI、DeepSeek 等）所驅動。使用能力較弱的模型，或透過會遮蔽或截斷上下文的中介平台路由請求，都會降低效能。此框架建立在創造人類負責人與底層模型原始能力之間最純粹對話的基礎上，確保每個代理都能以峰值潛能運作。

此框架的結構是一組專門代理團隊，每個代理都針對開發生命週期中的核心功能而設計。人類開發者作為中央協調者，委派任務並整合結果。

## 核心元件

為了有效運用前沿大型語言模型（large language model, LLM），此框架將不同開發角色指派給一組專門代理。這些代理不是獨立應用程式，而是透過精心設計、具角色特定性的提示與上下文，在 LLM 中喚起的概念人物設定（persona）。這種方法確保模型的龐大能力能精準聚焦於手上的任務，從撰寫初始程式碼到執行細緻且批判性的 review 皆然。

**協調者：人類開發者：** 在此協作框架中，人類開發者擔任協調者，作為 AI 代理的中央智慧與最終權威。

* **角色：** 團隊負責人、架構師與最終決策者。協調者定義任務、準備上下文，並驗證代理完成的所有工作。  
   * **介面：** 開發者自己的終端機、editor，以及所選 Agents 的原生 web UI。

**上下文暫存區：** 作為任何成功代理互動的基礎，Context Staging Area 是人類開發者細緻準備完整且任務特定 briefing 的地方。

* **角色：** 每項任務的專用工作區，確保代理收到完整且準確的 briefing。  
   * **實作：** 一個暫時目錄（task-context/），其中包含用於目標、code files 與 relevant docs 的 markdown files

**專家代理：** 透過使用有針對性的提示，我們可以建立一組 specialist agents，每個都為特定開發任務量身打造。

* **Scaffolder Agent：實作者**  
   * **目的：** 根據詳細規格撰寫新程式碼、實作功能，或建立 boilerplate。  
     * **Invocation Prompt：** "Y*ou are a senior software engineer. Based on the requirements in 01_BRIEF.md and the existing patterns in 02_CODE/, implement the feature...*"  
   * **Test Engineer Agent：品質守門人**  
     * **目的：** 為新程式碼或既有程式碼撰寫完整 unit tests、integration tests 與 end-to-end tests。  
     * **Invocation Prompt：** "You are a quality assurance engineer. For the code provided in 02_CODE/, write a full suite of unit tests using [Testing Framework, e.g., pytest]. Cover all edge cases and adhere to the project's testing philosophy."  
   * **Documenter Agent：書記**  
     * **目的：** 為 functions、classes、APIs 或整個 codebases 產生清楚、簡潔的文件。  
     * **Invocation Prompt：** "You are a technical writer. Generate markdown documentation for the API endpoints defined in the provided code. Include request/response examples and explain each parameter."  
   * **Optimizer Agent：重構夥伴**  
     * **目的：** 提出效能最佳化與程式碼重構，以改善可讀性、可維護性與效率。  
     * **Invocation Prompt：** "Analyze the provided code for performance bottlenecks or areas that could be refactored for clarity. Propose specific changes with explanations for why they are an improvement."  
   * **Process Agent：程式碼監督者**  
     * **Critique:** 代理會先進行初步檢查，識別潛在 bug、style violations 與邏輯缺陷，很像 static analysis tool。  
     * **Reflection:** 接著代理會分析自己的 critique。它會綜合發現、優先排序最關鍵的問題、略過過於吹毛求疵或低影響的建議，並為人類開發者提供高階、可行動的摘要。  
     * **Invocation Prompt：** "You are a principal engineer conducting a code review. First, perform a detailed critique of the changes. Second, reflect on your critique to provide a concise, prioritized summary of the most important feedback."

最終，這種人類主導的模型在開發者的策略方向與代理的戰術執行之間創造強大綜效。因此，開發者可以超越例行任務，把專業投入最能創造價值的創意與架構挑戰。

## 實務實作

### 設定檢查清單

若要有效實作人類－代理團隊框架，建議採用以下設定，重點是在提升效率的同時維持控制權。

1. **佈建前沿模型存取權** 為至少兩個領先大型語言模型取得安全的 API keys，例如 Gemini 2.5 Pro 與 Claude 4 Opus。這種雙供應商方法可用於比較分析，也能降低單一平台限制或停機的風險。這些 credentials 應像其他 production secret 一樣安全管理。  
2. **實作本機上下文協調器** 不要使用 ad-hoc scripts，而是使用輕量 CLI tool 或 local agent runner 來管理上下文。這些工具應允許你在專案根目錄定義一個簡單 configuration file（例如 context.toml），指定要將哪些 files、directories，甚至 URLs 編譯成單一 payload 供 LLM 提示使用。這可確保你對每次請求中模型看到的內容保有完整且透明的控制。  
3. **建立版本控制的提示程式庫** 在專案的 Git repository 中建立專用 /prompts 目錄。在其中將每個 specialist agent 的 invocation prompts（例如 reviewer.md、documenter.md、tester.md）儲存為 markdown files。把提示當作 code，可讓整個團隊隨時間協作、精煉並版本化提供給 AI 代理的指示。  
4. **將代理工作流程與 Git Hooks 整合** 使用 local Git hooks 自動化 review 節奏。例如，可以設定 pre-commit hook，在 staged changes 上自動觸發 Reviewer Agent。代理的 critique-and-reflection summary 可以直接呈現在你的終端機中，讓你在完成 commit 前獲得即時回饋，並把品質保證步驟直接嵌入開發流程。

### 領導增強式團隊的原則

若要成功領導此框架，開發者需要從個人貢獻者演進為人類－AI 團隊的負責人，並遵循以下原則：

* **維持架構所有權** 你的角色是設定策略方向並擁有高階架構。你定義「what」與「why」，並使用代理團隊加速「how」。你是設計的最終**裁定者**，確保每個元件都符合專案的長期願景與品質標準。  
* **掌握簡報的藝術** 代理輸出的品質直接反映其輸入品質。透過為每項任務提供清楚、明確且完整的上下文，掌握 brief 的藝術。不要把你的提示想成簡單命令，而應視為給一位新進且能力很強的團隊成員的完整 briefing package。  
* **擔任最終品質關卡** 代理的輸出永遠是提案，不是命令。把 Reviewer Agent 的回饋視為強大訊號，但你才是最終 quality gate。運用你的 domain expertise 與 project-specific knowledge 來驗證、質疑並批准所有變更，作為程式碼庫完整性的最終守護者。  
* **進行迭代式對話** 最好的結果來自對話，而不是獨白。如果代理的初始輸出不完美，不要丟棄它——精煉它。提供修正回饋、加入釐清上下文，並提示它再嘗試一次。這種反覆對話至關重要，尤其是對 Reviewer Agent 而言，其「Reflection」輸出被設計為協作討論的開始，而不只是最終報告。

## 結論

程式碼開發的未來已經到來，而且是增強式的。孤狼程式設計師的時代已讓位給新的典範：開發者領導由專門 AI 代理組成的團隊。此模型不會削弱人類角色；它透過自動化例行任務、擴大個人影響力，並達成過去難以想像的開發速度，提升了人類角色。

透過將戰術執行卸載給代理，開發者現在可以把認知能量投入真正重要的事情：策略創新、具韌性的架構設計，以及打造讓使用者喜愛的產品所需的創意問題解決。基本關係已被重新定義；這不再是人類與機器的競賽，而是人類巧思與 AI 之間的夥伴關係，像一支無縫整合的團隊共同運作。
